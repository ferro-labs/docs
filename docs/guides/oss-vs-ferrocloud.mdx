The open-source gateway is the engine that powers both editions. FerroCloud wraps it with multi-tenancy, a management dashboard, durable billing, and enterprise-grade security plugins.

## Security plugins (detail)

The five enterprise security plugins require FerroCloud because they depend on ML inference services that run server-side:

| Plugin | What it does | Requires |
|---|---|---|
| `pii-redact` | Detects and redacts PII (names, emails, phone numbers, SSNs…) using NER models | FerroCloud |
| `secret-scan` | Detects leaked API keys, tokens, and credentials using pattern + entropy analysis | FerroCloud |
| `prompt-shield` | Scores user messages for prompt injection attempts using a fine-tuned classifier | FerroCloud |
| `schema-guard` | Validates model JSON output against a JSON Schema before returning to client | FerroCloud |
| `regex-guard` | Blocks / warns on arbitrary regex rules (custom compliance patterns) | FerroCloud |

The OSS `word-filter` plugin provides simple keyword blocking without a cloud dependency.

## When to use the OSS gateway

- You need full control over infrastructure and data residency
- You are building a side project, prototype, or internal tool
- You want to self-host without per-seat licensing
- You are contributing to the open-source project

## When to use FerroCloud

- You need multi-tenant isolation (SaaS product built on top of Ferro)
- You need PII redaction, prompt injection protection, or secret scanning
- You want a managed dashboard with spend controls and team management
- You prefer SLA-backed infrastructure without DevOps overhead

## Data handling

Both editions process requests exclusively at runtime — model inputs and outputs are not stored by default. Enabling the `request-logger` plugin (OSS) or enabling logging in FerroCloud explicitly opts you into persistence. See [data handling](/security/data-handling) for details.
